# Campaigns

## Horizontal Campaigns

These are the main Genesis Mesh narratives.

| Campaign | Role |
|---|---|
| `why-genesismesh` | Problem framing: systems can connect, but cannot carry trust across boundaries. |
| `portable-trust` | Core thesis: trust should move between independent systems without one central owner. |
| `sovereign-systems` | Operator control: keys, policy, recognition, revocation, and trust decisions stay sovereign. |
| `recognition-and-revocation` | Trust lifecycle: it is not enough to trust once; you must cut trust off with proof. |
| `auditable-trust-state` | Evidence: every trust decision should be verifiable later. |
| `protocol-interoperability` | Protocol proof: Genesis Mesh is not tied to one runtime or language. |
| `trust-api-and-sdks` | Developer entry: stable Trust API and SDKs make the primitives usable. |

## Use Cases

These come after the horizontal campaigns.

| Campaign | Role |
|---|---|
| `use-cases/ai-agents` | Agents consume trust primitives; Genesis Mesh is not an agent framework. |
| `use-cases/edge-infrastructure` | Distributed infrastructure needs identity, revocation, and trust paths. |
| `use-cases/supply-chain-trust` | Maintainer attestations, release gates, and revocation after compromise. |
| `use-cases/enterprise-integration` | Enterprises keep local authority while recognizing external trust. |
| `use-cases/data-access-governance` | Data access carries intent, policy, evidence, and verification. |
